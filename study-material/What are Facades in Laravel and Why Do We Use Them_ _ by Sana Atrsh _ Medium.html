What are Facades in Laravel and Why Do We Use Them?

Sana Atrsh

·

Follow

3 min read
·
Aug 27, 2023

Many of beginner don’t now too much about Facades, even though it’s one of the must important architecture concepts in Laravel and understanding it will help to make more powerful and professional projects.

So, What is Facades?

As a Laravel programmer you have seen Route:: in web.php file. Well, this is a facade!

Facades provide a static interface to classes that are available in the application’s service container (which is a place to store variables and objects to use them when we need). Which mean that we can use functions without making an object from class.

let’s see this file in Laravel project vendor\laravel\framework\src\Illuminate\Support\Facades\Auth.php where Auth:: facade is. We can see this function:

 protected static function getFacadeAccessor()
    {
        return 'auth';
    }

Laravel’s facades, and any custom facades you create, will extend the base Illuminate\Support\Facades\Facade class.

Notice that near the top of the file we are “importing” the Auth facade. This facade serves as a proxy for accessing the underlying implementation of the Illuminate\Support\Facades\Authinterface. Any calls we make using the facade will be passed to the underlying instance of Laravel's authentication service.

The function getFacadeAccessor() returns auth , and The name of this facade is being returned so we can access the named facade anywhere within the Laravel application without initializing it. So we can do something like Auth::user()->nameanywhere really easily.

Why Do We Use Facades?

As you can see, facades make code faster to write, easier to read and more organized.

Imagen if you have to remember Illuminate\Auth\AuthManager or another 40 classes instend of just Auth:: or any another facade name!

here’s the list of facades in Laravel documentation.

to understand that more, let’s use this example:

use Illuminate\Auth\AuthManager;
 
public function index()
{
    $auth = new AuthManager();
    $user = $auth->user();
}

here, to use AuthManager in $user (which give $user authentication functions) we first create an object $auth from AuthManager class.

When we use Auth:: facade, it become like this:

use Illuminate\Support\Facades\Auth;

public function index()
{
    $user = Auth::user();
}

This is a very simple example. But when we use it in a real project with a more facades it become way shorter.

Why (Sometimes) We Should Not Use Facades?

In bigger Laravel projects, creating complex classes with a lot of facades can cause a scope creep. Since facades are so easy to use, it can be easy to let your classes continue to grow and use many facades in a single class.

So, when using facades, pay special attention to the size of your class so that its scope of responsibility stays narrow. If your class is getting too large, consider splitting it into multiple smaller classes.

So as we saw, facades is a powerful feature in Laravel. This was a very simple introduction to facades, please let me know what do you think!
